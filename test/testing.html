Project - Automated Identification of Data Constraints from Software Documentation

Problem description and motivation

Software systems are crafted in adherence to guidelines and constraints outlined by business rules. Data constraints dictate permissible or non-permissble values for the data managed within systems. A data constraint has one or more operands, corresponding to the entity/property and the value. For example, the underlined text in the following sentence, “If you specify a repeat count greater than 20, a confirmation dialog box will be displayed”, describes the constraint: repeat count > 20, which has the operands repeat count and 20. Data constraints can be categorized into four types:  In the example above, it falls under the value comparison type. For details on data constraint operands and types, please refer to the paper [1].

Figure 1. Examples of data constraints (highlighted)

These data constraints typically originate from the domain of context of the issues the software aims to tackle. Developers consequently are tasked with coding to ensure these constraints are upheld within the system. Previous research has studied how data constraints are implemented in Java and how to trace them to their implementations in Java code [1-2]. However, extracting data constraints from software system in prior work was done manually, which might limit the applications of data constraints. Additionally, previous work revealed that manually identifying and labeling data constraints from software documents written in natural language is a time-consuming and error-prone task for developers. Therefore, this project aims to automatically identify data constraints from software documentation in natural language.

Preliminary research

We manually labeled 546 data constraints from nine Java systems, finding that data constraints are described in nine discourse patterns. These manually labeled data constraints are considered as positive instances. Additionally, we collected tool-generated sentence fragments that are semantically equivalent to those manually labeled data constraints, which form another  set of positive instances. On the contrary, we collected 25,808 sentence fragment as negative instances. All instances have five descriptions: text, sentence that contains the text, constraint operands, constraint type and discourse patterns.

We will explore the suitability of traditional machine learning techniques, utilizing five descriptions as features for automatically classifying fragments as data constraint descriptions. Two types of evaluation—10-Fold Cross-Validation and Cross-System Evaluation—are required. To tackle the imbalanced dataset, where the number of constraints is much less than the tool-generated fragments (i.e., 546 versus 26,641), we may need to employ four additional approaches, which are duplicating data constraints, sampling negative fragments, hybrid over/under-sampling and intelligent over-sampling techniques.

Steps

High level idea of the steps: Given a dataset containing positive instances that describe a data constraint (manually labeled/tool-generated) and negative instances that do not describe a data constraint, we need to conduct two types of evaluation (code will be provided) to determine the best configuration for classifying sentence fragment instances as data constraints.

Running experiments by doing the following:

  1. Read the related work [1-3] to understand data constraints, constraint operands, types, and discourse patterns.
  2. Go through the provided dataset, code instructions, and Python source code to grasp the logic behind the classifier and learn how to configure the code.
  3. Follow our instructions to execute classifiers for both 10-Fold Cross-Validation and Cross-System Evaluation in order to obtain preliminary classification results, including precision, recall, and F1 score.
  4. Please send the following data: the executed configuration and file name, execution logs, and a summary of your findings to yzhou29@gmu.edu for further analysis.

Data labeling

Access the data: constraints

In there are 1304 tool-generated fragments that share common terms with data constraints. Two coders manually checked these fragments and identified equivalent relationships between 627 fragments and 547 data constraints. A tool-generated fragment and a data constraint are considered equivalent if they meet the following criteria: 1) they must be located in the same sentence, 2) they must share the same discourse pattern and constraint type, and 3) they must have semantically equivalent operands.

Evaluation

Goal: evaluate the performance of the classifier in a “real-world” usage scenario

We used a 90-10 stratified split: 90% for training and 10% for testing sentences.
